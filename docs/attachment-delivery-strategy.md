# 附件投递与按需 OCR 策略

## 目标

用户上传附件后，系统先保存稳定的本地快照路径，再根据当前模型的 `inputModalities` 决定是否额外提供原生输入。OCR 不在上传或发送时自动执行，而是作为模型可按需调用的只读工具。

## 模态策略

`provider-catalog.yaml` 目前只描述模型原生输入能力：`text`、`image`、`video`。

- 所有附件都会进入模型文本上下文中的附件清单，清单包含文件名、类型、大小和快照路径。
- 图片附件在模型支持 `image` 时会额外转成 Base64 原生图片输入；不支持 `image` 时只提供路径。
- PDF 不属于原生输入模态，始终只提供路径，不自动 OCR，也不自动渲染页图。
- 视频当前只提供路径。虽然 YAML 可以声明 `video`，但现有 shared run 契约只支持 `image/*`，后端 pi 模型链路也只实际发送 `text/image`。
- 文本、代码和轻量 Office 类附件保留现有抽文本上下文，同时也会出现在附件清单中。

## OCR 工具

后端提供 `OcrExtractText` 只读工具，支持图片和 PDF，不支持视频，也不会修改、压缩或转换文件。工具只有在当前 run 的可见附件中存在图片或 PDF 时才对模型可见。

工具调用时会记录：

- 请求路径与解析后的目标路径
- PDF 起始页与页数限制
- 文件类型、总页数、已处理页数
- OCR 文本长度、耗时和 warning 数量

OCR 模型资源在工具第一次调用时懒加载，避免上传大附件时阻塞发送流程。

## 典型场景

- 大 PDF 压缩：模型从附件清单拿到 PDF 路径，直接使用文件或 shell 工具处理，不触发 OCR。
- 文本模型读图片文字：模型看到图片路径和 OCR 提示后，按需调用 `OcrExtractText`。
- 多模态模型看图片：系统同时提供图片路径和 Base64 原生图片输入，模型可直接看图。
