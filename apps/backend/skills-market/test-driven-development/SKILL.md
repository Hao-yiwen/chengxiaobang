你正在帮助用户以**测试驱动开发（TDD）**的方式编写代码：先写测试，再写实现，小步循环。

## 铁律

**在拥有一个失败的测试之前，不写任何一行生产代码。**

这条规则没有例外。不是"先写实现再补测试"，不是"这个太简单不用测"，而是：测试先行，永远先行。如果你发现自己正在没有失败测试的情况下修改生产代码，立刻停下，删掉改动，回去写测试。

## 循环：RED → GREEN → REFACTOR

### 1. RED——写一个最小的失败测试

- 一次只测**一个行为**。测试名要说清这个行为："空输入时返回错误"，而不是"测试函数"。
- 测试要小：理想情况下几行断言就够。如果测试需要大量铺垫，说明被测单元设计有问题，先记录下来。
- 写完后**必须运行它**，用基础 shell 能力执行项目的测试命令（如 `pnpm test`、`npx vitest run <文件>`、`python -m pytest <文件>`）。

### 2. 确认它因"正确的原因"失败

- 失败信息必须是断言失败（期望 X 实际 Y），而不是导入错误、语法错误或测试框架配置问题。
- 如果失败原因不对，先修测试本身，直到它精准地因"功能还不存在/行为不正确"而失败。
- 这一步是在**校验测试自身的有效性**——一个因错误原因失败的测试，将来也无法保护你。

### 3. GREEN——用最少的代码让它通过

- 只写让当前这个测试通过所需的代码，不多写一行。不顺手实现"将来会用到"的分支。
- 哪怕实现看起来"笨"也没关系，优雅留给下一步。
- 运行测试，确认这个测试通过，且**没有破坏其他已有测试**（跑全量测试套件）。

### 4. REFACTOR——在绿灯下重构

- 测试全绿后，才可以消除重复、改善命名、提取函数。
- 重构期间不改变任何外部行为，不新增功能。
- 每做一小步重构就跑一次测试，保持绿灯。

然后回到第 1 步，为下一个行为写下一个失败测试。

## 常见借口 vs 现实

| 借口 | 现实 |
| --- | --- |
| "这个功能太简单了，不用测" | 简单代码的测试也很简单，写它只要一分钟；而"简单代码"恰恰是低级 bug 的高发地 |
| "我先写完实现，最后统一补测试" | 事后补的测试只会验证"代码现在的行为"，而不是"应该有的行为"，等于给 bug 盖章 |
| "测试拖慢了我的进度" | 慢的是调试，不是测试。没有测试时你靠手工反复验证，总耗时远高于一次写好测试 |
| "这只是个原型/脚本" | 原型有惊人的概率直接进生产。如果它真是一次性的，用户会明确说 |
| "已有代码没有测试，我跟着不写也合理" | 你写的每个测试都在改善现状；跟随坏习惯只会让代码库继续腐烂 |
| "UI/IO 不好测" | 把逻辑抽成纯函数就好测了。不好测本身就是设计反馈 |

## 修 bug 时的 TDD

1. 先写一个**能复现该 bug 的失败测试**——它精确描述"正确行为应该是什么"。
2. 运行确认失败，失败现象与用户报告的 bug 一致。
3. 修复实现，让该测试通过。
4. 跑全量测试确认无回归。这个测试将永久防止此 bug 复发。

## 完工核对清单

收工前逐条确认：

- [ ] 每一个新行为都有对应测试，且该测试**曾经失败过**（你亲眼见过 RED）。
- [ ] 全量测试套件通过，用基础 shell 能力跑了完整命令，不是只跑了新文件。
- [ ] 没有任何"先跳过、回头补"的测试（无 `skip`/`todo` 残留，除非用户同意）。
- [ ] 实现中没有测试未覆盖的分支逻辑（投机写的代码要么删掉，要么补测试）。
- [ ] 测试断言的是行为和输出，而不是内部实现细节（避免一重构就碎一片）。

只要有一条不满足，工作就没有完成。
