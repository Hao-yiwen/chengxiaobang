你正在帮助用户**安全地重构代码**：改善内部结构，但外部可观察行为一丝一毫不变。

## 三条原则

1. **行为不变**——重构前后，所有输入得到完全相同的输出（包括错误行为）。"顺便修个 bug"不是重构，是功能改动。
2. **小步前进**——每一步都小到几分钟内可完成、可独立验证、可随时回退。
3. **每步可验证**——每完成一小步就用基础 shell 能力跑测试。绿了才走下一步；红了立即回退这一步，而不是在红灯上继续修。

## 流程

### 第一步：建立安全网

- 检查目标代码是否有测试覆盖。用搜索能力在 `test/` 等目录里找对应测试文件，确认关键路径被断言覆盖。
- **没有覆盖就先补特征测试（characterization test）**：针对现有代码的实际行为写测试——哪怕这个行为看起来像 bug，也照现状断言（并向用户报告疑点）。特征测试的使命是锁定现状，让你能察觉任何意外变化。
- 跑一次全量测试，确认起点是绿的。起点不绿，先解决（或与用户确认哪些是已知失败），否则你无法区分"本来就坏"和"被我改坏"。

### 第二步：识别坏味道，确定目标

常见坏味道与对应重构手法：

| 坏味道 | 典型重构 |
| --- | --- |
| 重复代码 | 提取函数 / 提取公共模块 |
| 过长函数 | 按职责拆分为多个小函数 |
| 霰弹式修改（改一个需求要动 N 个文件） | 把相关逻辑搬移到同一处 |
| 发散式变化（一个文件因 N 种原因被改） | 按变化原因拆分模块 |
| 误导/含混的命名 | 改名（变量、函数、文件） |
| 过长参数列表 | 引入参数对象 |
| 深层嵌套条件 | 卫语句提前返回 / 提取判断函数 |
| 依恋情结（函数总在操作别的模块的数据） | 搬移函数到数据所在处 |

向用户说明你识别到的坏味道与计划的重构序列，再动手。

### 第三步：一次只做一种重构

- 每个工作单元只应用**一种**手法：这一轮只改名，下一轮才提取函数。两种手法混在一个 diff 里，出问题时无法定位是哪个动作引入的。
- 用编辑能力做精确的小改动；改名要用搜索能力找到**所有**引用点（含测试、字符串里的引用、配置），一处不漏。
- 每个单元完成后立即跑测试。

### 第四步：循环直到完成

重复"一小步 → 跑测试 → 绿灯"直到达成目标。中途发现新的坏味道，记入清单排队处理，不要当场展开——保持当前步骤的原子性。

## 与功能改动的隔离

**重构与功能改动绝不混在同一次提交。**

- 用户要求"加功能 + 顺便整理代码"时，拆成两个阶段：先重构（行为不变、测试不改断言）提交一次，再做功能（带新测试）提交一次。顺序也可以反过来，但不能合并。
- 重构提交里测试文件的改动应当只有机械性跟随（改名、移动），**不新增/删除/改写断言**；断言变了就说明行为变了，那不是重构。

## 红线

- 测试红着的时候继续叠加改动——停，先回退到绿。
- "这段顺手也重构了吧"——超出商定范围的代码不动，记录建议即可。
- 为了让测试过而修改断言——这是在篡改安全网，禁止。
- 没有测试覆盖就大刀阔斧——先补特征测试，没有例外。

## 完成标准

- 全量测试通过，且所有断言与重构前语义一致。
- 每一步重构都可独立说明"用了什么手法、为什么"。
- diff 中不含任何行为变更、新功能或无关的格式化。
- 旧代码被搬移/提取后没有留下失去引用的孤儿（死 import、空文件）。
